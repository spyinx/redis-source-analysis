## 漏洞概述

Redis 的 `BITFIELD` 和 `BITFIELD_RO` 命令支持 `#<offset>` 语法，其中偏移量会按位域宽度自动缩放。当传入极大的 `#<offset>` 值时，乘法运算 `loffset * bits` 在 `long long` 有符号整数上发生**溢出**，触发**未定义行为（Undefined Behavior）**。

在启用了 **UBSAN**（Undefined Behavior Sanitizer）或加固编译的环境中，这种溢出直接导致 **Redis 服务器进程异常终止**，形成**远程可触发的拒绝服务（DoS）攻击**。

## 影响版本

- 需回退修复至 **Redis 8.8 Backport**
- 已合并至 `unstable` 分支

## BITFIELD 命令基础

### 命令作用

`BITFIELD` 把 Redis 字符串当成**位域数组**（bit array）来操作：
- **GET** — 读取特定位置、特定宽度的整数
- **SET** — 修改特定位置的整数
- **INCRBY** — 原子性递增

常用于紧凑存储多个小整数、位图统计等场景。

### 基本语法

```bash
# 在偏移 0 处读取 8 位有符号整数
BITFIELD mykey GET i8 0

# 在偏移 8 处设置 16 位无符号整数为 1000
BITFIELD mykey SET u16 8 1000

# 原子递增
BITFIELD mykey INCRBY u8 0 1
```

### 两种偏移语法

| 语法 | 含义 | 示例 |
|------|------|------|
| `offset` | 从第 offset 个**位**开始 | `GET i8 0` → 第 0 位开始读 8 位 |
| `#offset` | 从第 offset 个**元素**开始，自动乘宽度 | `GET i8 #3` → 第 3×8=24 位开始 |

### 漏洞触发点

```
GET i64 #144115188075855872  →  offset = 144115188075855872 × 64
                                = 超过 LLONG_MAX → 有符号整数溢出
```

### 修改文件

### `src/bitops.c` 详细改动

**修改前（存在漏洞）**：
```c
if (string2ll(p+usehash, plen-usehash, &loffset) == 0) {
    addReplyError(c, err);
    return C_ERR;
}

/* 乘法在检查之前 —— 溢出点 */
if (usehash) loffset *= bits;

/* 范围检查在乘法之后 —— 太晚了 */
if (loffset < 0 || (!mustObeyClient(c) && (loffset >> 3) >= server.proto_max_bulk_len))
```

**修改后（已修复）**：
```c
/* ✅ 修复1：解析后立即检查负数 */
if (string2ll(p+usehash, plen-usehash, &loffset) == 0 || loffset < 0) {
    addReplyError(c, err);
    return C_ERR;
}

if (usehash) {
    /* ✅ 修复2：乘法前溢出检查 */
    if (loffset > LLONG_MAX / bits) {
        addReplyError(c, err);
        return C_ERR;
    }
    loffset *= bits;
}

/* ✅ 修复3：移除重复负数检查 */
if (!mustObeyClient(c) && (loffset >> 3) >= server.proto_max_bulk_len)
```

### 修复逻辑

| 修复点 | 说明 |
|--------|------|
| 提前拒绝负数 | `string2ll()` 后立刻检查 `loffset < 0` |
| 乘法前溢出检查 | `loffset > LLONG_MAX / bits` 时拒绝 |
| 移除冗余检查 | 后续条件中删除 `loffset < 0` |
| 统一错误返回 | 所有违规返回 `ERR bit offset is not an integer or out of range` |

## 问题复现

### 环境准备

```bash
# 编译带 UBSAN 的 Redis（修复前版本）
make -C src \
    SANITIZER=undefined \
    MALLOC=libc \
    OPTIMIZATION=-O0 \
    PROG_SUFFIX=-ubsan \
    redis-server-ubsan redis-cli-ubsan
```

### 复现命令

```bash
# 启动 UBSAN 服务器
./src/redis-server-ubsan --port 6379 --bind 127.0.0.1 \
    --save '' --appendonly no --daemonize yes

# 触发溢出（以下任一命令均可）
./src/redis-cli-ubsan BITFIELD_RO k GET i64 '#144115188075855872'
./src/redis-cli-ubsan BITFIELD k GET i64 '#144115188075855872'
```

### 修复前预期现象

- 服务器进程异常终止
- 日志中出现：
  ```
  bitops.c:730:26: runtime error: signed integer overflow:
  64 * 144115188075855872 cannot be represented in type 'long long int'
  ```
- 后续 `PING` 命令无响应

## 安全影响评估

| 维度 | 评估 |
|------|------|
| **远程可利用性** | ✅ 是，任何能连接 Redis 的客户端均可触发 |
| **权限要求** | 仅需能执行 `BITFIELD` 或 `BITFIELD_RO` |
| **影响范围** | 进程级 DoS，服务器崩溃 |
| **数据影响** | 未持久化数据可能丢失 |
| **利用难度** | 极低，单条命令即可触发 |

### 修复前缓解措施

1. **禁用 UBSAN**：生产环境通常不使用 UBSAN 构建
2. **ACL 限制**：限制不可信用户对 `BITFIELD`/`BITFIELD_RO` 的访问
3. **网络隔离**：避免 Redis 直接暴露到公网

---

## 总结

PR #15433 修复了一个中等风险的有符号整数溢出漏洞。修复方案简洁有效：
- 解析偏移量后立即拒绝负值
- `#<offset>` 乘法前增加溢出检查 `loffset > LLONG_MAX / bits`
- 统一返回已有错误信息

此修复已合并至 `unstable`，并计划回退至 `Redis 8.8`。
